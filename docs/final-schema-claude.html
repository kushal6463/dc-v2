Business-rooted causal KG — tri-axis Domain ∥ Product ∥ Platform, master-config-free

Ten node labels ship V1, anchored on a single Business root with an equal tri-axis spine — Domain (the FRD's functional columns), IntelligenceProduct (the IQ apps), and Platform (vendor source/action systems — thin & lazily materialized) — all feeding the Metric hub, which denormalizes domain + product + platform onto itself. Governance (Policy/Threshold) is defined now and populated later. RBAC is seniority-clearance + domain-branch, fed by an adaptive Org Graph Engine. Every master-config dependency is removed.

Verdict Codex sourced identity/edges/governance from stale master-config and dropped the product/domain nodes — both disqualifying. The Claude draft kept the nodes but rooted the spine on product, contradicting the FRD, and lacked a Business root and real seniority RBAC. This doc keeps the Claude backbone, fixes the topology to the FRD, and cuts master-config entirely.

RBAC — seniority clearance + domain branch + adaptive Org Graph Engine

Build the allowed context first, then answer from it. VIEW is gated by clearance + branch; EDIT/APPROVE is a separate FRD owner/approval gate.

graph-native
VIEW gate — a role may view a resource iff both:
(1) Clearance  role.seniority_rank ≥ resource.min_level
(2) Branch  resource's domain ∈ the role's domain branch (home domain_id ∪ domain_scope_ids[] ∪ domains reachable down REPORTS_TO) or an explicit CAN_VIEW {allow} cross-grant — and no higher-priority CAN_VIEW {deny}.
EDIT / APPROVE is separate. Seeing a metric never implies editing its threshold. Edit/approve follow explicit CAN_EDIT/CAN_APPROVE edges, approval_required, and the approval path derived from owner_role_ids (e.g. Marketing Director → Creative Lead → Finance → Execution Owner).

Subordinate blocked

A GM-level feature has min_level = 70. A Manager (rank 50) under the GM fails clearance → can't see it.

Manager capped

A board-only metric has min_level = 95. The GM (rank 70) fails clearance → masked even in-branch.

Sees level & below

A VP of Marketing (rank 80, branch marketing) sees marketing resources ≤ 80 — but not Finance internals without a cross-grant.

Org Graph Ingestion Engine — adaptive & editable

The API has no org-hierarchy data, and org shape varies (a startup has no CFO/CMO; an MNC has a full C-suite). A separate LLM service produces the social graph as proposals — never a direct write.

1 · InputBusiness.tier + org description (free text / HR export / titles / headcount)
2 · ProposeLLM digests, scaled to tier → Role nodes + REPORTS_TO + seniority_rank + domain ownership
3 · Arbitrateone writer dedupes/validates → writes Role subgraph (is_engine_generated = true)
4 · Edithumans add/remove/re-rank as the org grows; edits are sticky and survive re-runs
REPORTS_TO (org/social tree — branch test + escalation) and INHERITS_FROM (permission inheritance) are kept distinct. A role may inherit permissions it does not socially report through, and vice versa.

Allowed-metrics query (clearance + branch baked in)

MATCH (r:Role {role_key: $role_key, status: 'active'})
OPTIONAL MATCH (r)-[:REPORTS_TO*0..6]->(:Role)-[:OWNS|OWNED_BY]->(:Domain)   // branch via org tree
WITH r, collect(DISTINCT r.domain_id) + r.domain_scope_ids AS branch_domains
MATCH (m:Metric {status: 'active'})
WHERE r.seniority_rank >= m.min_level                                        // (1) clearance
  AND ( ANY(d IN m.domain_ids WHERE d IN branch_domains)                   // (2a) branch — m.domain_ids is an array
        OR EXISTS { MATCH (r)-[:CAN_VIEW {effect:'allow'}]->(m) } )         // (2b) cross-grant
  AND NOT EXISTS { MATCH (r)-[:CAN_VIEW {effect:'deny'}]->(m) }
RETURN DISTINCT m.metric_uid, m.concept_name, m.product_ids, m.domain_ids, m.min_level;

Domain-edge write convention ⊕ — every metric/dashboard gets exactly one spine edge BELONGS_TO_DOMAIN to its home column (the first/primary id in domain_ids[]). The edge is truth; the cache is rebuilt from it.

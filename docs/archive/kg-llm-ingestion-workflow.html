Ingestion workflow

How the LLM
builds the graph

A deterministic pre-pass feeds parallel Claude proposer agents, which classify and link metrics into JSON proposals. A single writer commits them after review — then a causal pass adds evidence-backed edges. Here is every stage, and exactly where the model is — and isn't — involved.

⊘

The LLM never writes to Neo4j.

It only proposes nodes and edges as JSON. Every write goes through one sequential arbitration writer after review. Combined with canonical_id uniqueness constraints, duplicate nodes and write races are structurally impossible. The model's job is classification, formula reading, link resolution, and causal judgment — not the write.

Architecture at a glance

One pipeline, three kinds of work

Pure code does the heavy structural extraction. The LLM runs read-only in the middle. A single deterministic writer is the only thing that touches the database.

Step by step

The seven stages

Each stage is color-coded by who does the work. Commands are the real kg CLI; every one has a Claude Code slash-command twin.

STEP 0

One-time setup

pure code

Install, apply Neo4j constraints, and seed the spine. The spine defines the only IDs the LLM is later allowed to link to — it can never invent a domain or product.

# install, constraints, seed Business / Domain / Product spine
uv sync --extra dev
uv run kg schema-init
uv run kg bootstrap-spine
uv run kg status

Slash twins: /create-business-node · /create-domain-node · /create-product-node

STEP 1

Deterministic pre-pass

pure code · 0 tokens

The biggest design decision: structural fields are extracted in plain code so the LLM context stays tiny.

  • Joins openapi.json (~902 paths) with chart-registry.json (646 entries) on {dashboard}:{id}.
  • Drops excluded endpoints — master-config, /auth, POST/DELETE, health/docs.
  • Emits one deterministic draft dict per metric & dashboard.
uv run kg prepass            # ~600 metric + dashboard drafts
uv run kg prepass --json

harness/ingest/prepass.py

STEP 2

Proposer agents — the LLM generates nodes & edges

LLM · read-only

One Claude sub-agent per dashboard, fanned out concurrently (default 6). Each runs with subscription OAuth, schema-constrained JSON output, and setting_sources=[] — no project hooks, MCP, or skills.

  • Reads only a tiny RAG shortlist of spine IDs from Neo4j (the single read; strictly read-only).
  • Reconciles metrics into shared concept_key; classifies with allowed enums only.
  • Reads each formula_text → proposes DECOMPOSES_INTO edges (conf 1.0).
  • Picks chart_type; emits VISUALIZES · SHOWN_ON · BELONGS_TO_DOMAIN · PART_OF_PRODUCT.
  • Output is merged onto the deterministic draft — "draft → agent reconciles."
uv run kg ingest-dashboard ceo-pulse
uv run kg ingest-dashboard ceo-pulse --auto-approve
uv run kg ingest-all --limit 5 --concurrency 6

harness/ingest/proposer.py · harness/agent/engine.py · prompts.py

STEP 3

Review the proposals

human · canvas

Proposals land with review_state:"proposed". Nothing is in Neo4j yet. A reject re-spawns that dashboard's proposer with the reason injected as steering (Reflexion-style).

uv run kg proposals list --run <run_id>
uv run kg proposals approve <id> --all
uv run kg proposals reject <id> --reason "..."
STEP 4

Arbitration writer — the only thing that writes Neo4j

single writer

Sequentially consumes approved proposals with idempotent MERGE … ON CREATE … ON MATCH …. Run it twice → identical counts. Every write appended to the events.jsonl replay log.

uv run kg apply --run <run_id>

harness/kg/arbitration.py

STEP 5

Cross-partition reconcile

pure code

A metric that appeared on N dashboards is collapsed into one node, preserving every dashboard's relationships. Unused spine nodes are pruned.

uv run kg reconcile          # --dry-run to preview
uv run kg prune-empty

harness/kg/reconcile.py

STEP 6

Causal pass — guarded LLM influences

LLM · review-only

Mostly deterministic. The LLM is used only for INFLUENCES candidates, and they are never auto-committed.

  • DECOMPOSES_INTO / ROLLS_UP_TO — deterministic, confidence 1.0.
  • CORRELATES_WITH — imported priors; never auto-promoted to CAUSES.
  • INFLUENCES — statistical gate first → pointwise judge (empty mechanism = reject) → refuter → self-consistency → Beta(α,β) confidence.
  • Confidence is the agreement fraction, not the model's stated number (verbalized confidence is mis-calibrated).
uv run kg run-causal                  # deterministic only
uv run kg run-causal --llm            # + review-only INFLUENCES
uv run kg run-causal --llm-links --platform meta_ads

harness/ingest/causal.py

Why it's safe

Design guarantees

No hallucinated nodes

The agent can only link to spine IDs it was given; the output schema rejects off-vocabulary values.

No write races or dups

A single sequential writer + canonical_id uniqueness make them structurally impossible.

No auto-committed causality

Correlations never become CAUSES; every influence carries confidence · evidence_mass · mechanism and is held for review.

Calibrated confidence

Causal confidence is the self-consistency agreement fraction — not the model's stated number.

Flat context window

Pre-pass + per-dashboard partitioning + schema output + RAG shortlist keep every call small and cheap.

Fully replayable

Every write is appended to events.jsonl; Neo4j is durable across restarts.

Final KG direction

Business-rooted graph with product, domain, and platform axes

The final schema replaces the flat six-label model with a KG-only graph that supports multiple intelligence products, business decision domains, and source/action platforms under one business root.

Claude comparison update: Codex remains canonical, with Claude's stronger field typing, master-config-free metric identity, defined-only governance, seniority/domain-branch RBAC, and rollup/decomposition edge semantics folded in.
Platform decision: keep platform/source fields on Metric always. Materialize Platform as a thin node when platform ownership, RBAC, freshness, outage, grouping, or future action routing needs traversal.
Business IntelligenceProduct Domain Platform (lazy) Metric Dashboard UIComponent Policy Threshold Role
Product axis

Business -> IntelligenceProduct -> Dashboard -> UIComponent -> Metric.

Domain axis

Business -> Domain -> Metric for ownership, governance, and decisions.

Platform axis

Business -> Platform -> Metric only when platform traversal is materialized.

Detailed Data Model

Every KG record should be explicit about identity, ownership, source, status, and refresh behavior. Edges are source of truth for traversal; denormalized arrays on Metric are fast-read caches.

Data areaImplementation standard
Stable identityEvery node has a deterministic id that does not change when display names or routes change.
Source ownershipNodes and relationships carry source metadata or are created by an arbitration writer that records source metadata externally.
Review stateHarvested or inferred records can be proposed or needs_review; only reviewed records become active.
Status over deletionMissing or retired records become deprecated, hidden, blocked, or archived; ingestion must not delete them automatically.
Cached contextMetric arrays are rebuilt from graph edges. If an edge and a cache disagree, the edge wins.
Access classificationSensitive resources must declare data_classification and min_level, or inherit them from product/domain/policy defaults.

Role And Social Graph

Access requires seniority clearance, domain branch or explicit cross-grant, data classification clearance, and no explicit deny.

access_allowed =
  role.seniority_rank >= resource.min_level
  AND (resource is in the role's domain branch OR an explicit cross-grant applies)
  AND role.max_data_classification allows resource.data_classification
  AND no explicit deny edge applies

The Org Graph Engine can generate proposed role hierarchy and domain-branch edges, but it is planning infrastructure for schema/RBAC population, not a decision runtime being implemented now.

Source And Ingestion Rules

Automated ingestion must not use stale master-config/** sources. They are never node truth, edge truth, identity truth, governance truth, or secondary evidence.

Excluded automated sources

master-config/**
master-config/config/metrics
master-config/config/dashboards
master-config/config/thresholds
master-config/config/knowledge-graph/relationships

Allowed sources

  • Business-plane OpenAPI endpoint families outside master-config/**.
  • Live dashboard/card/chart endpoints outside excluded master-config/** surfaces.
  • Current app route metadata.
  • Chart registry for UI component inventory and chart explanations.
  • Connector/platform manifests where available.
  • Deterministic formula parsing.
  • Reviewed manual config.
  • Statistical or LLM proposals routed through review.

Build Order

1Constraints

Create constraints and indexes.

2Business

Seed the root node.

3Axes

Seed products and domains; materialize platforms when traversal is enabled.

4Surfaces

Load dashboards and components.

5Metrics

Load canonical metrics with cached axis fields.

6Edges

Create product, domain, optional platform, surface, and metric edges.

7Roles

Seed role and social graph edges from reviewed Org Graph Engine and role-map proposals.

8Governance

Create Policy and Threshold schema shells with population_status = defined; do not ingest instances yet.

9Causal

Add causal edges from reviewed evidence/proposals only; do not import relationship catalogs from master-config.

10Defer

Keep decision runtime for later.

Acceptance Criteria

  • The new KG has a Business root.
  • One Business can map to many IntelligenceProduct nodes.
  • Domain is a first-class decision axis.
  • Platform/source data is always available on Metric; Platform is a thin lazily materialized node when platform-level traversal is needed.
  • Metric identity uses metric_uid, canonical_id, and metric_id with distinct meanings.
  • Metrics are canonical and not duplicated per product/dashboard/platform.
  • Detailed node data is defined for business, product, domain, optional platform, metric, dashboard, component, policy, threshold, and role records.
  • Policy and Threshold are defined-only V1 shells with population_status; no instances are harvested from master-config.
  • Relationship properties are defined for traversal, governance, source, causal, rollup/decomposition, threshold, and permission edges.
  • Ingestion proposals are emitted and reviewed before graph writes.
  • Dashboard ids are not sourced from excluded master-config/** surfaces.
  • Role access uses seniority_rank, domain branch or explicit cross-grant, data_classification clearance, and no explicit deny.
  • rare_seeds appears only as pilot/client historical context.
  • Decision runtime and memory layers are deferred.
  • Current implementation scope is KG-only.
Final shape: Business -> IntelligenceProduct -> Dashboard -> UIComponent -> Metric, Business -> Domain -> Metric, and Business -> Platform -> Metric when platform traversal is materialized.

ThoughtWire Causal Knowledge Graph

A metric surface, rebuilt as a signed causal graph.

dc‑kg turns dashboards, metrics, formulas, ML models and governance rules into a directed knowledge graph in Neo4j — populated by a fleet of Claude agents, written through a single schema‑enforcing gate, and explored on a live React Flow canvas.

01

Overview

▾

A flat metric catalog can list numbers; it can't answer the questions that actually drive decisions. The graph can:

QuestionGraph capability
If this warehouse column changes, which metrics break?lineage / blast radius (source_columns scan)
What drives ROAS, and with what lag and sign?signed causal upstream / downstream traversal
What is this metric made of?DECOMPOSES_INTO formula decomposition
Who governs this metric and what alerts on it?Policy / Threshold governance overlay

What changed — skeletal → agentic

The project began as a deterministic "skeleton" build (and earlier, an in-memory NetworkX prototype). Those were scope-blind resolving formulas and additivity-blind rolling metrics up. The system was rebuilt around an LLM agentic builder (kg build): bounded-concurrent Claude Opus agents read the curated catalog and backend SQL, then create nodes and draw edges through one schema-enforcing writer. The skeleton CSVs survive under data/ as inputs and regenerable artifacts — no longer the build mechanism. (Their historical design docs now live in docs/archive/.)

Source-of-truth / input files

▾

The graph is constructed from a small set of curated inputs. Every node and edge traces back to one of these — nothing is invented from nothing.

Spine seeds — harness/seed/ (the deterministic backbone)

FileContains
spine_seed.jsontri-axis spine: business, domains[], products[] → Business / Domain / IntelligenceProduct
platforms.jsonPlatform nodes (ga4, google_ads, meta_ads, klaviyo, magento…) + sub-channels
component_types.jsonthe 17 generalized chart-type UIComponent nodes
governance.rare_seeds.json · governance_seed.pydemo Policy + Threshold + governance edges and their loader
concept_causal_rules · funnel_flow · identities · *_overrides · *_priorsenrichment priors/overrides consulted during build + enrich

Metric catalog — data/ (the metric source of truth)

FileContains
metric_nodes.rare_seeds.jsonThe metric source of truth. _meta + 325 metrics (uid, canonical_id, formulas, deps, ML flags, scope) + 161 input_nodes. This is what Phase-1 agents read.
metric_registry.rare_seeds.csvactively read during the build — OpenAPI registry: formula, source_expr, mart_model, dashboards. The doc-tool get_metric_source joins this row; enrich.py reads it for SQL/mart provenance.
unique_metrics_catalog.rare_seeds.csvdeduplicated, sub-platform-aware metric manifest
*_edges.rare_seeds.csvNOT consumed by the build. structural / compositional / crossproduct / model are dead skeleton outputs (no code reads them); candidate / discovered are I/O for the optional kg discover engine only. Edges are derived by the agents (§05), not imported.

External backend — BC_2 (read-only)

The sibling dbt repo. Agents read its seed CSVs and SQL only through MCP doc-tools (get_metric_source, get_bc2_sql, inspect_bc2_sources) — never by writing into it. harness/ingest/bc2_snapshot.py hashes + caches the relevant slice.

Proposal queue — data/proposals/run-*/

Dashboard-ingestion output: one JSONL per dashboard, each line a review-state proposal (node + relationship payloads). Reviewed, then apply-ed through arbitration.

05

The AI / LLM layer

▾

Three LLM callsites — all Claude Opus 4.8, all writing only through the single arbitration writer.

5.1 · Agentic build — kg build (the main one)

Four phases, parallel within a phase, with a hard barrier before any edge is drawn:

phase 1
Nodes
~8–12 agents read their metric slice via doc-tools & create_metric_node, attaching to the spine.
phase 2
Structural
DECOMPOSES_INTO edges from formulas via draw_edge.
phase 3
Causal
INFLUENCES edges from reasoning over metrics + BC_2 — confidence, sign, lag.
phase 4
Critique
single agent finds loops / orphans / leaves, de-dupes, writes the build report.
— barrier · asyncio.gather · every node exists before edges —

kg build --dry-plan previews the plan fully offline (no SDK, no Neo4j). --smoke builds only the blended.* ROAS chain and skips the destructive wipe. A full build is destructive: Phase 0 exports a backup, wipes, and re-seeds the spine first.

5.2 · Dashboard ingestion proposer

harness/ingest/proposer.py turns one dashboard's charts into proposals (review-before-write) against a section-8 output schema — not direct graph writes.

5.3 · Governance extraction

harness/governance/extract.py parses a governance document into draft Policy/Threshold fields to prefill the canvas authoring wizard.

Why this is safe

single writer sandboxed tools (no shell/web/fs) idempotent MERGE + audit events

08

Node & edge model

▾

Node labels — 10 defined, 9 materialized

Business Domain IntelligenceProduct Platform Metric Dashboard UIComponent Policy Threshold Role reserved Chart synthetic · view-only
⚠ Role
Schema-reserved, not materialized in V1. Role has full schema support — uniqueness constraints (role_id, role_key), a seniority_rank index, and a Pydantic model. But no code creates Role nodes: there is no create_role_node tool, no seed emits one, and the canvas palette renders the nine materialized labels (plus the synthetic Chart view node). RBAC is modeled in the schema but deferred. Treat Role as reserved.

Metric.node_kind further classifies a metric node as metric / intermediary / input / constant.

▦ Chart
Synthetic view node — frontend-only, never in Neo4j. Charts are not a graph node type (their fields are folded onto Metric / the chart-registry per the M2 decision). The canvas renders a Chart node (provenance: "synthetic", id chart::<canonical_id>) only on shift-click, hydrated from /api/metric-chart or /api/dashboard-charts and wired by a RENDERED_BY (metric→chart) or SHOWN_ON (chart→dashboard) synthetic edge. It is excluded from search / apply / write paths (FR-CG-008).

Edge model — metric ↔ metric is exactly two types

DECOMPOSES_INTO — structural: formula · component · identity · rollup · crossproduct · funnel
INFLUENCES — causal: curated_rule · promoted · llm_verified · statistical · statistical_candidate
inverse hop — a denominator/subtrahend component (sign −1)
cross-domain — spans two metric domains
deprecated — never deleted, marked status='deprecated'

Spine / surface / governance edges: HAS_DOMAIN, HAS_PRODUCT, USES_PLATFORM, PART_OF_PRODUCT, SHOWN_ON, VISUALIZES, GOVERNS, ENFORCES_THRESHOLD, HAS_THRESHOLD. Edges carry an evidence ledger (Beta confidence), review_state, source_kind, and a status lifecycle.

Frontend — app/kg-canvas

▾

A React 19 + Vite SPA. Graph rendering uses React Flow (@xyflow/react) with a dagre layered layout and an ego/focus layout. State is a single Zustand store; styling/legends are centralized in lib/graphTheme.ts; API + SSE wrappers in lib/api.ts.

Layout shell (App.tsx): CommandSearch + Toolbar + ProgressBar on top; CanvasView fills the center; a left governance drawer and a right tabbed inspector (Activity · Review · Node · Edge · Edge Diff) overlay the canvas.

▶
The frontend is read-mostly: it visualizes the graph and drives the review/approve and governance-authoring workflows. It never writes the graph directly — every mutation goes through the API → arbitration. (POST /api/run-causal is retired; construction is kg build only.)

Overview modes & shift-click drill-down (graphLayout.ts + store.ts)

Four overview layouts: Spine (metrics under their Domain — default), Tree (decomposition forest), Map (hub skeleton), and Dashboards — the curated main dashboards (dashboard_type ∈ {executive, review} or id ending -overview) clustered under their Product (dashboardGroupedLayout). Filtering NODE KINDS to a non-spine label (Dashboard / Policy / Threshold / Chart) now packs those nodes into a grid instead of blanking (the Spine layout falls back when it finds no spine nodes).

Shift-click drill-down dispatches by node kind and reveals synthetic, client-only view nodes/edges (never persisted — FR-CG-008), which the focus ring clusters around the clicked node:

  • Metric → revealMetricChart — its specific Chart (RENDERED_BY), auto-selected for ChartDetail.
  • Dashboard → revealDashboardCharts — every chart on it (Chart -[SHOWN_ON]-> Dashboard, via /api/dashboard-charts) plus its real SHOWN_ON metrics.
  • IntelligenceProduct → revealProductDashboards — that product's dashboards (Dashboard -[PART_OF_PRODUCT]-> Product).

Charts are surfaced this way because they are not a graph node type; the synthetic builder (makeChartNode) merges the chart-registry slice with metric props, keyed by canonical_id so the same chart dedupes whether reached via its metric or its dashboard.

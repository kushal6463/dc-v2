V1 Graph Shape

A tree-like navigation spine for orientation; a temporal causal DAG for truth; governance and RBAC ride as overlays — connected, never melted in.

8 labels · 4 layers
Spine & structureProduct→Domain→Metric orientation; dashboard composition; visualization.
GovernancePolicy, threshold, and the enforcement link between them.
CausalMetric-to-metric influence, correlation, reviewed causality — with temporal lag.
RBACRole permissions filter context before the agent sees it.

Not a tree

The spine is for orientation only; the causal model is a temporal DAG where feedback loops resolve as spirals down the time axis.

Edges are the project

rare_seeds has 355 nodes and only 4 edges (all correlation, one spurious). Nodes are solved; trustworthy causal edges are the entire job.

Confidence is derived

V1 edges carry a simple confidence; the append-only evidence ledger that derives it is V2.

Two-Tier RBAC

The app authenticates the user, resolves role_key, then builds the allowed context first — never an unrestricted traversal filtered afterward.

filter before context

Tier 1 — product gate

Role -CAN_ACCESS_PRODUCT-> IntelligenceProduct. No MIQ entitlement ⇒ marketing metrics invisible.

Tier 2 — domain/metric

Role -CAN_VIEW-> Domain/Metric/Dashboard. A causal path crossing into restricted territory returns a masked fact, never the name/value.

Edge-naming note. Codex used per-action labels (CAN_VIEW_METRIC, CAN_EDIT_THRESHOLD…). This finalised schema keeps generic labels with a permission property (CAN_VIEW, CAN_EDIT) — it scales better and keeps queries uniform. Either is valid; do not mix them.

Allowed context for one metric

MATCH (r:Role {role_key: $role_key, status: 'active'})
MATCH (m:Metric {metric_uid: $metric_uid, status: 'active'})
WHERE EXISTS {
  MATCH (r)-[grant:CAN_VIEW]->(m)
  WHERE grant.effect = 'allow'
}
OPTIONAL MATCH (m)-[:HAS_THRESHOLD]->(t:Threshold {status:'active'})
OPTIONAL MATCH (m)-[:GOVERNED_BY]->(p:Policy {status:'active'})
OPTIONAL MATCH (c:UIComponent)-[:VISUALIZES]->(m)
OPTIONAL MATCH (d:Dashboard)-[:RENDERS]->(c)
OPTIONAL MATCH (up:Metric)-[:CAUSES]->(m)
RETURN m, collect(DISTINCT t), collect(DISTINCT p),
       collect(DISTINCT c), collect(DISTINCT d),
       collect(DISTINCT up.metric_uid) AS upstream;

Safe edit check

MATCH (r:Role {role_key: $role_key, status: 'active'})
MATCH (t:Threshold {threshold_id: $threshold_id, status:'active'})
MATCH (r)-[grant:CAN_EDIT]->(t)
WHERE grant.effect = 'allow'
RETURN t.threshold_id,
       grant.approval_required AS approval_required;

Role closure

MATCH (r:Role {role_key: $role_key, status:'active'})
OPTIONAL MATCH (r)-[:INHERITS_FROM*0..4]->(p:Role {status:'active'})
RETURN collect(DISTINCT r)+collect(DISTINCT p) AS role_scope;

The causal layer starts empty and is built from scratch — no legacy relationship catalog is imported. Raw statistical associations may enter as CORRELATES_WITH but are never auto-promoted to CAUSES. V1 success is measured in evidence-backed causal edges, not node count.

Deferred to V2 (named, not specified)

Out of scope for lean V1. The FRD's memory/learning machinery (Layers 2–6) plus the Layer-1 entities both prior drafts omitted. Listed so nothing is lost — deliberately without property tables. Defer until the spine, RBAC, governance, and a real causal layer are live, or memory pollutes causal truth.

V2 candidates

Causal-graph entities the FRD names (Layer 1)

Outcome (result distinct from a metric) · Tool/Action (CONTROLLED_BY; google_ads_budget, meta_budget) · Investigation_Rule (harness-enforced "must-check" that blocks a recommendation if data is missing) · Approval_Rule (approval path derived from node owners).

Memory / learning / governance (Layers 2–6)

Thoughtlet · DecisionCapsule (10-section, ANCHORED_TO nodes for reverse temporal context injection) · MonitoringContract + WakeCondition (time + event hooks) · EvidenceEvent + CausalRelation (Beta-fold derived confidence) · LearningCandidate→PromotedMemory · GraphChangeProposal · GraphVersion (time-travel) · SourceProfile/Endpoint_Family/IngestionRun · Capsule-Agent rehydration recipe · Person.

These were what the Claude draft modelled in its Part II and what the Codex draft omitted entirely. They are the actual ThoughtWire differentiator (decision→monitor→learn) — V2's job, not V1's.
